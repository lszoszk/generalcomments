## Critical (fix before next push)

**C1 ┬╖ 30 of 77 Playwright tests fail on `main`**
Full chromium run: **47 passed / 30 failed / 6 skipped (9.8 min)**. Same
8 failures reproduce on the committed baseline (`git stash` round-trip
confirmed) тАФ every failure is **pre-existing**, not a v19.50.1 regression.
Three root-cause clusters:

- **bootApp timeout** (D1, D8, F3, F5, F9, F10, several smoke + report
  specs) тАФ `_helpers.ts` waits тЙд15 s for `mast-folio` text + `body
  [data-active-view]`. On this machine the corpus + JUR-lite fetch
  often blows that budget. Either raise the wall to 30 s, or split
  the predicate so the test reports *which* signal is missing.
- **API-mode dependency** (A1тАУA9, UX2, UX4) тАФ tests assume
  `https://150.254.115.204/unhrdb-api` is reachable from the runner.
  When the VM is down (or the test machine is offline), every API
  test fails. Suite needs an `?api=0` opt-out path or a mocked API
  fixture.
- **UI-text drift** (smoke 1/3/7/8/9, W3) тАФ assertions like
  `/DOCUMENTS$/`, `Keep typing`, `.dossier-toolbar` no longer match
  the rendered DOM. Production-side: those assertions are stale.
  Update them to the current copy/selectors.

The pre-push hook runs the suite, so this is currently bypassed for
every push via `UNHRDB_SKIP_TESTS=1`. **Net effect: the hook is a
placebo until the suite is green.**

**C2 ┬╖ One critical axe violation on documents view**
`aria-required-children`, 1 node. Likely the `tablist` (GC/JUR/SP scope
picker) is missing `role="tab"` on its children. Quick fix.
Source: `tests/a11y-audit.spec.ts` (temp script during this audit, removed).

**C3 ┬╖ Race in three async functions тАФ no `runId` re-check after `await`**
Pattern `if (runId !== state.searchRun) return;` is used in
`runSearch` but missing in:
- `runSearchViaApi` (`app.js:254-283`) тАФ 7 state writes after the
  network await with no re-check; a second keystroke during flight
  paints stale results over fresher state.
- `openDocReader` (`app.js:1683-1709`) тАФ clicks A then B before A's
  shard load finishes тЖТ A's continuation clobbers B's `docsActive*`.
- `jumpToParagraph` (`app.js:7462-7479`) тАФ JUR load can complete
  after the user has navigated away; still calls `setView('search')`.

## High (next batch)

**H1 ┬╖ Suspect "merged-article" values in JUR data (5 distinct, ~6 occurrences)**
Beyond the 6 truly bogus values fixed by `_docs_internal/sanitize_articles.py`
(`Art. 319450`, `Art. 761879`, etc.), 5 more are mechanically wrong:
`Art. 95`, `Art. 85`, `Art. 65`, `Art. 61`, `Art. 51`. ICCPR has 53
articles, so anything > 53 in a CCPR case is impossible. These are
almost certainly `Art. 9(5)`, `Art. 8(5)`, etc. with the parentheses
stripped during ingest. Recovering the original token is impossible
without re-extracting from source PDFs тАФ recommend a manual pass for
those 5 cases or extending the sanitiser threshold from `> 99` to
`> 53` (would silently drop them, which is at least non-misleading).

**H2 ┬╖ 279 JUR docs have bare-string `optionalProtocolArticlesParsed` entries**
Mixed shape: most entries are `{instrument, article, paragraph}`
objects, but 557 entries across 279 docs are bare strings like
`'OP1 Art. 2'`. The dossier renderer assumed objects тЖТ printed
`Art. undefined` chips. Fixed in v19.50.1 by coercing strings to
the rendered label, but the data should be normalised in the next
ingest batch so the runtime guard isn't load-bearing.

**H3 ┬╖ 3,054 JUR records use prose adoptionDate (`25 October 2010`)**
Not ISO 8601. 96% of JUR docs. Not currently a runtime bug because
the date is rendered raw, but it breaks date-arithmetic features
(year-range slider, sort-by-date) for everything except the
`adoptionYear` integer field. Normalise to `YYYY-MM-DD` in the next
ingest pass.

**H4 ┬╖ 173 GC docs have NO `status` field set**
Only 4 statuses populated across 187 GCs (`final`, `revised`,
`superseded` on 14 docs). The remaining 173 are `null`, which means
the runtime "Hide superseded GCs" toggle hides 2 docs but the *correct*
universe is all docs whose status is `superseded`. Most of those 173
are likely `final` and just never tagged. One row of fact-checking on
the OHCHR catalogue would close this.

**H5 ┬╖ Search query terms highlighted in dossier but NOT in reader**
`paintDossier` calls `renderParagraphHtml(text, fns, { terms })`
(`app.js:5685`); `paintDocReaderBody` calls
`renderParagraphHtml(text, fns)` with **no terms**
(`app.js:1885`). User opens a paragraph from search тЖТ highlights in
dossier preview disappear when they click "Open in document reader".
Surprising for a research workflow.

**H6 ┬╖ 14 serious color-contrast violations on the search view**
axe flags 14 nodes failing WCAG 2.0 AA contrast on the search view
alone, 6 each on documents/workspace. Mostly `.dim` text and the
`.badge-preview/.badge-jur` chips on the cream background. Bumping
those colors is a 5-line CSS edit.

**H7 ┬╖ CDN scripts loaded without Subresource Integrity (SRI)**
`xlsx.full.min.js` (`app.js:3154`) and `flexsearch.bundle.min.js`
(`app.js:3464`) тАФ both over HTTPS but no `integrity=` hash. A CDN
compromise injects arbitrary JS into the SPA. Add SRI hashes and an
`onerror` that disables the feature gracefully.

## Medium (code health)

**M1 ┬╖ 5 functions > 200 lines**
`paintDossier` 613, `bindUI` 340, `paintDocReaderBody` 230,
`runSearch` 200, `_buildEmptyState` borderline at 87. Each is a
plausible split target.

**M2 ┬╖ State-mutation hot spots without setters**
- `state.activeId` mutated from тЙе7 sites
- `state.filters.*` Sets mutated by тЙе10 chip handlers (each duplicates
  toggle + URL persistence + `runSearch()` plumbing)
- `state.docsActiveDocId/ParaId` written from 5+ sites

A single `toggleFilter(category, value)` and a `setActive(paraId)`
helper would consolidate three classes of "did you remember to call
the second helper?" bugs.

**M3 ┬╖ Duplicated rendering paths**
Citation popover rendered three times (drawer line 2096, dossier
footer line 5759, inline popover line 6282); workspace-mark buttons
rendered twice (`renderResult` 5078 vs `refreshResultMarks` 5283);
`escapeRegex` and `escapeRe` are identical regexes with two names.
Pick one source of truth per cluster.

**M4 ┬╖ Boot try/catch swallows step-level failures**
`boot()` (`app.js:512-639`) wraps the entire init sequence тАФ manifest
fetch, all paints, every binding, the post-paint `runSearch()` тАФ in
one `try/catch`. Any failure prints "Failed to load" with no info on
*which* step crashed. Sprinkle `try` blocks around the network calls
that legitimately fail; let painters throw upstream.

**M5 ┬╖ Empty `catch {}` in clipboard handlers**
`copy paragraph` (line 5973) and `copy citation` (line 6305) catch and
discard the clipboard write error, then show a "Copied!" toast even
when the write threw. Show a real error toast on rejection.

**M6 ┬╖ Dead code**
- `initCompactHeader` (`app.js:7744-7773`) тАФ boot at 574-575 explicitly
  doesn't call it; ~30 lines unreferenced.
- No-op delegated-handler loop comment at `paintDocsRail`
  (`app.js:1601-1607`) тАФ supposed to be deleted in a follow-up that
  never landed.
- Obsolete TODO at line 198 ("Sprint-3.1: chain pages on scroll" тАФ
  already shipped via `fetchNextApiPage`).

**M7 ┬╖ 19 `console.*` calls with inconsistent prefixes**
Mostly `console.warn` for legitimate errors. One `console.info`
("FlexSearch built in N ms") fires every cold boot тАФ surprising for
end users opening DevTools. Prefixes vary (`[unhrdb-api]`, `[jur shard]`,
`[tour]`, plain text). Pick one.

## Medium (perf)

**P1 ┬╖ Boot pulls 11.4 MB of `documents-lite.json` synchronously**
Everything else is lazy (corpus 25 MB after first paint, JUR shards
~3-5 MB on demand). The lite catalog blocks the first render тАФ but
on closer look it's loaded in `Promise.all` with the smaller files,
so the bottleneck is purely network + parse. Worth profiling whether
splitting it into per-committee shards gives a meaningful perceived-
boot win.

**P2 ┬╖ 245 functions in one 8,113-line file**
No module split. For a static SPA with no build step this is a deliberate
choice (and the `defer`-able post-paint pattern is well-tuned), but
the file is past the "comfortable to navigate by symbol" threshold.
ESM split into ~6 modules (search, docs-reader, dossier, filters,
data-loaders, tour) would not cost runtime тАФ `<script type="module">`
already supports multi-file dynamic imports тАФ and would shrink the
diff surface for every change.

## Low (cosmetic / maintainability)

**L1 ┬╖ 140+ "v19.X-fixN" version-drift comments**
Many describe behaviour that's been stable for several versions. A
one-pass cleanup of comments referencing versions older than v19.40
would reduce reading load substantially.

**L2 ┬╖ Cache-buster strategy is manual**
`?v=20260507-v19-50` updated by hand in three places in `index.html`.
A 5-line build script (or a bash one-liner in the pre-push hook) could
stamp this from `git rev-parse --short HEAD`.

**L3 ┬╖ No CSP**
`docs/index.html` ships no Content-Security-Policy meta. With the SRI
gap (H7) and 61 innerHTML sites, a CSP with `script-src 'self'
cdn.sheetjs.com cdn.jsdelivr.net googletagmanager.com; object-src
'none'` would close one tier of XSS impact even if a future bug landed
on the page.

**L4 ┬╖ `docs_v2/` and `_docs_internal/` are untracked but live in repo**
Both directories contain real files (some referenced by ingest tooling).
Decide: track them or `.gitignore` them. Currently they're invisible
to anyone cloning the repo.
